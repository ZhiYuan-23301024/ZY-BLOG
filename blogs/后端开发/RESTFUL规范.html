    RESTFUL规范

    ZhiYuan 11/1/2025

    在 RESTful API 的设计规范中，不同类型的 HTTP 请求（方法）在是否使用请求体（Body）以及请求体的限制方面，主要遵循的是 HTTP 协议本身的语义和广泛的实践约定。下面的表格概括了不同请求类型的特点和限制。

    请求方法 主要用途 请求体使用建议 关键限制因素
    ​GET​ 检索/查询资源 ​不推荐使用。根据规范，GET 请求的参数应通过 URL 的查询字符串（Query String）传递。 ​URL 长度。受浏览器和服务器配置限制，通常建议将总长度控制在 2KB 以内，否则可能被截断或返回 414 错误。
    ​DELETE​ 删除资源 ​通常不使用。要删除的资源标识符应放在 URL 中。幂等操作。 ​HTTP 语义。虽然协议未禁止，但携带请求体可能被某些服务器实现忽略，因此应避免。
    ​POST​ 创建资源/提交数据 ​推荐使用。请求体是提交数据（如表单、JSON）的主要方式。非幂等操作。 ​服务器配置。理论无上限，但实际受服务器（如 Nginx, Tomcat）的 max-body-size等设置限制，可从几 MB 到几 GB。
    ​PUT​ 完整更新资源 ​推荐使用。请求体应包含资源的完整新表示。幂等操作。 ​服务器配置。同 POST 请求，限制主要来自服务器端的配置。
    ​PATCH​ 部分更新资源 ​推荐使用。请求体应包含描述资源如何被修改的指令（如 JSON Patch）。 ​服务器配置。同 POST 和 PUT 请求，限制主要来自服务器端的配置。

    # 💡 实际应用建议

    了解上述规范后，在实际开发和设计接口时，可以遵循以下建议以确保API的健壮性和良好实践：

    • ​参数传递选择​：对于创建（POST）、更新（PUT/PATCH）等操作，将复杂数据放在请求体（Body）中，通常使用 application/json格式。对于获取（GET）或删除（DELETE）操作，使用URL路径（如 /users/123）或查询字符串（如 ?id=123）来标识目标资源。

    • ​处理大量数据​：当需要传输的数据量非常大时（例如上传大型文件），不要依赖单个巨大的请求体。更可靠的做法是使用分片上传技术，将大文件分割成多个小块进行传输。

    • ​明确配置服务器​：务必在服务器端（如Nginx、Spring Boot、Express等）明确设置请求体大小的上限，并返回清晰的错误信息（如413 Payload Too Large），以便于前端进行错误处理。

    希望这份概括能帮助你更好地理解 RESTful 规范下的请求体使用。如果你对某个特定场景有更具体的问题，我很乐意提供更深入的分析。
