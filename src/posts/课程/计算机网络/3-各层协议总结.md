## 应用层协议

其实我觉得可以通过判断“信息是不是有用户请求语义”来判断是不是应用层协议，网络层通常没有包含用户意图，但是应用层有，比如用户想知道怎么配置IP，用户想获取超文本内容，网络层比较机械，只在乎数据包的传输，但并不在乎数据包里用户的意图

| 协议         | 端口号             | 依赖协议       | 说明                                                                 |
| :--------- | --------------- | ---------- | ------------------------------------------------------------------ |
| HTTP/HTTPS | 80/443          | TCP        | **超文本传输协议**，用于网页浏览，HTTPS是加密版本，基于TLS/SSL。**也就是说，在TCP的基础上又加了一层加密层。** |
| FTP        | 20/21           | TCP        | 文件传输协议，用于文件的上传和下载。20为数据通道，21为控制通道。                                 |
| SMTP       | 25              | TCP        | 简单的邮件传输协议，用于邮件的发生                                                  |
| POP3       | 110/995         | TCP        | 邮局协议，用于从邮件服务器接收邮件，995是加密版本                                         |
| DNS        | 53              | UDP（部分TCP） | 域名系统，将域名解析为IP地址。较大响应或区域传输时使用TCP                                    |
| DHCP       | 67（服务器）/68（客户端） | UDP        | 动态主机配置协议，自动分配IP地址和其它网络参数                                           |

```ad-note
title:应用层协议扫盲

- 超文本
	带有超链接的文本。何谓超链接，传统的文本是线性阅读的，但是由于超链接（hyperlink）的存在，我可以跳跃阅读。其实超文本在传输的角度看也是纯文本，只不过部分文本有链接语义罢了。**超文本协议是为了web请求资源设计的，包括如何请求如何响应，只不过web请求的资源大多数是超文本资源。**

- 我又发现一个协议叫作邮件传输协议SMTP，我觉得很扯淡，为什么要有这个专门的协议，直接用http传输文本，然后在软件层面设计格式不就好了吗
	**历史惯性，当时没有通用的应用层协议，每个应用都自己造轮子**。且SMTP有它区别于其它协议的地方，如何断点续传？如何离线缓存？都是要做的，当然，这些基于http+软件程序控制也可以做到，只不过SMTP作为一个协议更专业
```
